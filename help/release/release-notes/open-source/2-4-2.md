# Note sulla versione di Magenti Open Source 2.4.2

Magenti Open Source 2.4.2 introduce miglioramenti a livello di prestazioni e sicurezza oltre a miglioramenti significativi della piattaforma. I miglioramenti della sicurezza includono l&#39;espansione del supporto per i `SameSite` attributo per tutti i cookie. Elasticsearch 7.9.x e Redis 6.x sono ora supportati.

Questa versione include oltre 280 nuove correzioni al codice core e 35 miglioramenti della sicurezza. Include la risoluzione di quasi 290 questioni GitHub da parte dei nostri membri della community. Questi contributi della community spaziano dalla pulizia secondaria del codice di base ai miglioramenti significativi in GraphQL.

Tutti i problemi noti identificati nella versione 2.4.1 sono stati risolti in questa versione.

## Applica `AC-3022.patch` continuare ad offrire DHL come vettore marittimo

DHL ha introdotto la versione dello schema 6.2 e disattiverà la versione 6.0 dello schema a breve. Adobe Commerce 2.4.4 e versioni precedenti che supportano solo la versione 6.0 dell’integrazione DHL. Si applicano i merchants che distribuiscono queste versioni `AC-3022.patch` al più presto per continuare ad offrire DHL come vettore marittimo. Consulta la sezione [Applicare una patch per continuare ad offrire DHL come vettore di spedizione](https://support.magento.com/hc/en-us/articles/7707818131597-Apply-a-patch-to-continue-offering-DHL-as-shipping-carrier) Articolo della Knowledge Base per informazioni sul download e l&#39;installazione della patch.

## Patch di sicurezza disponibile

Gli esperti di marketing possono ora installare correzioni di sicurezza che richiedono tempi di esecuzione, senza applicare le centinaia di correzioni funzionali e i miglioramenti forniti da una versione trimestrale completa (ad esempio, 2.4.1-p1). La patch 2.4.0.12 (pacchetto compositore 2.4.1-p1) è una patch di sicurezza che fornisce correzioni per le vulnerabilità che sono state identificate nella nostra versione trimestrale precedente, 2.4.1. Tutte le hotfix che sono state applicate alla versione 2.4.1 sono incluse in questa patch di sicurezza. A *hotfix* fornisce una correzione a una versione rilasciata che risolve un problema o un bug specifico.)

Per informazioni generali sulle patch di sicurezza, vedi [Introduzione alla nuova versione della patch di sicurezza](https://community.magento.com/t5/Magento-DevBlog/Introducing-the-New-Security-Patch-Release/ba-p/141287). Per istruzioni su come scaricare e applicare le patch di sicurezza (compresa la patch 2.4.1-p1), vedi [Installazione rapida in locale](../../../installation/composer.md). Le patch di sicurezza includono solo correzioni di bug di sicurezza, non i miglioramenti di sicurezza aggiuntivi inclusi nella patch completa.

## Altre informazioni sulla versione

Anche se il codice per queste funzioni è in bundle con le versioni trimestrali , diversi di questi progetti (ad esempio, Progressive Web Application (PWA) Studio) vengono rilasciati anche in modo indipendente. Le correzioni di bug per questi progetti sono documentate nelle informazioni sulla versione separate e specifiche per il progetto, disponibili nella documentazione di ciascun progetto.

## Elementi di rilievo

Cerca le seguenti funzioni di rilievo in questa versione.

### Miglioramenti sostanziali della sicurezza

Questa versione include oltre 35 correzioni di sicurezza e miglioramenti alla sicurezza della piattaforma. Tutte le correzioni di sicurezza sono state riportate in 2.4.1-p1 e 2.3.6-p1.

#### Oltre 35 miglioramenti a livello di sicurezza per la chiusura di vulnerabilità RCE (Remote Code Execution) e XSS (cross-site scripting)

Finora non si sono verificati attacchi confermati relativi a tali problemi. Tuttavia, alcune vulnerabilità possono essere potenzialmente sfruttate per accedere alle informazioni sui clienti o per prendere il controllo delle sessioni di amministrazione. La maggior parte di questi problemi richiede che un autore dell&#39;attacco ottenga prima l&#39;accesso all&#39;amministratore. Di conseguenza, ti ricordiamo di prendere tutte le misure necessarie per proteggere il tuo amministratore, compresi, tra l’altro, questi sforzi: inserire nell&#39;elenco Consentiti IP, [autenticazione a due fattori](https://devdocs.magento.com/guides/v2.4/security/two-factor-authentication.html), l&#39;utilizzo di una VPN, l&#39;utilizzo di una posizione unica anziché `/admin`e una buona igiene delle password. Vedi [Bollettino sulla sicurezza di Adobe](https://helpx.adobe.com/security/products/magento/apsb21-08.html) per una discussione su questi problemi fissi.

#### Ulteriori miglioramenti della sicurezza

I miglioramenti di sicurezza per questa versione includono:

* Tutti i cookie di base ora supportano la funzione `SameSite` attributo.

* L’applicazione ora visualizza messaggi che identificano contenuti potenzialmente dannosi nei campi di descrizione di prodotto e categoria quando l’utente tenta di salvare i valori in questi campi.

* Le operazioni del file system tra i vari componenti sono state standardizzate e consolidate per evitare caricamenti dannosi.

* Sono state corrette le violazioni all’informativa sulla sicurezza dei contenuti di base (CSP).

### Miglioramenti all&#39;infrastruttura

Questa versione contiene miglioramenti alla qualità di base, che migliorano la qualità del Framework e delle seguenti aree funzionali: Account cliente, Catalogo, CMS, OMS, Importazione/Esportazione, Promozioni e Targeting, Carrello e Pagamento.

### Miglioramenti alla piattaforma

* [**È ora supportato Elasticsearch 7.9.x**](../../../installation/system-requirements.md). Anche se si consiglia di eseguire Elasticsearch 7.9.x, la versione 2.4.x rimane compatibile con Elasticsearch 7.4.x. <!-- MC-36867-->

* 2.4.2 è stato testato con [Vernice 6.4](../../../installation/system-requirements.md). La versione 2.4.x rimane compatibile con Varnish 6.x.

* [**È ora supportato Redis 6.x**](../../../installation/system-requirements.md). La versione 2.4.x rimane compatibile con Redis 5.x. <!-- MC-34853-->

* 2.4.2 è ora compatibile con **Compositore 2.x**. Si consiglia ai commercianti di migrare a Composer 2.x. Sebbene sia possibile installare questa versione utilizzando Composer 1.x, Composer 1.x terminerà presto. Per una panoramica delle funzioni di Composer 2.x, vedi [Compositore 2.0 è ora disponibile!](https://blog.packagist.com/composer-2-0-is-now-available/)

In questa versione, la possibilità di configurare un’installazione per l’utilizzo di un database suddiviso è stata rimossa. I commercianti che al momento utilizzano il database diviso dovrebbero iniziare a pianificare il ripristino o la migrazione a un singolo database o utilizzare un approccio alternativo. Consulta la sezione [Funzionalità di database suddiviso nel Magento Open Source obsolete](https://community.magento.com/t5/Magento-DevBlog/Deprecation-of-Split-Database-in-Magento-Commerce/ba-p/465187) Post DevBlog per una panoramica di questo problema.  Vedi [Ripristino da un database diviso a un singolo database](../../../configuration/storage/revert-split-database.md) per istruzioni sulla migrazione.

### Miglioramenti delle prestazioni

Questa versione include miglioramenti al codice che migliorano le prestazioni API e il tempo di risposta dell’amministratore per le distribuzioni con cataloghi di grandi dimensioni. I miglioramenti a livello di scalabilità multiplo consentono alla versione 2.4.2 di supportare cataloghi complessi fino a 20 volte più grandi rispetto alle versioni precedenti.

### Integrazione Adobe Stock

Questa versione include l&#39;integrazione di Adobe Stock v2.1.1.

### GraphQL

Questa versione aggiunge una copertura GraphQL per le seguenti funzioni:

* È stato aggiunto il supporto per [liste di confronto](https://devdocs.magento.com/guides/v2.4/graphql/queries/compare-list.html). Gli acquirenti possono [creare](https://devdocs.magento.com/guides/v2.4/graphql/mutations/create-compare-list.html) e [delete](https://devdocs.magento.com/guides/v2.4/graphql/mutations/delete-compare-list.html) elenchi di confronto e [add](https://devdocs.magento.com/guides/v2.4/graphql/mutations/add-products-to-compare-list.html) e [remove](https://devdocs.magento.com/guides/v2.4/graphql/mutations/remove-products-from-compare-list.html) elementi negli elenchi di confronto. Inoltre, gli acquirenti che creano un elenco di confronto come ospite possono effettuare l&#39;accesso come cliente e [conservare](https://devdocs.magento.com/guides/v2.4/graphql/mutations/assign-compare-list-to-customer.html) le loro liste di confronto.

* È stato aggiunto il [`generateCustomerTokenAsAdmin`](https://devdocs.magento.com/guides/v2.4/graphql/mutations/generate-customer-token-as-admin.html) mutazione e aggiornamento del `Customer` per supportare l&#39;assistenza agli acquisti remota.

* È stato aggiunto il supporto per la localizzazione in diversi store per supportare attività quali la modifica di lingue, carrelli e valute. <!-- MC-37801-->

* È stato aggiunto il supporto per i sindacati in GraphQL. [GitHub-29425](https://github.com/magento/magento2/issues/29425) <!-- ENGCOM-8291 -->

* Lo schema GraphQL è stato migliorato per ottimizzare il recupero dei dati di prodotto per prodotti configurabili con molte varianti. <!-- MC-36138-->

* Gli ID degli oggetti di tipo intero sono stati dichiarati obsoleti a favore di `uid` attributi di tipo ID. <!-- MC-36346-->

* È stato aggiunto il `staging` attributo [`ProductInterface`](https://devdocs.magento.com/guides/v2.4/graphql/interfaces/product-interface.html) e [`CategoryInterface`](https://devdocs.magento.com/guides/v2.4/graphql/interfaces/category-interface.html) per determinare se un prodotto è in staging e visualizzare le informazioni sulla campagna associate. <!-- MC-33897-->

Consulta la sezione [Guida per gli sviluppatori di GraphQL](https://devdocs.magento.com/guides/v2.4/graphql/) per informazioni dettagliate su questi miglioramenti.

**Nuove risorse per il ruolo di Media Gallery**. Questa versione offre agli esercenti la possibilità di limitare l’accesso dell’amministratore solo alla raccolta multimediale e di controllare chi può eseguire queste azioni:

* Inserire risorse multimediali nel contenuto

* Caricare le risorse

* Modifica dei dettagli delle risorse

* Eliminare le risorse da Media Gallery

* Gestire la struttura delle cartelle.

**Immagini ottimizzate per il web nel contenuto**. Gli esercenti ora possono utilizzare [rendering di immagini ottimizzate per il web](https://docs.magento.com/user-guide/cms/media-gallery-image-optimization.html) in contenuti invece di immagini ad alta risoluzione. L&#39;immagine originale rimane non modificata in Media Gallery e il rendering dell&#39;immagine viene generato in modo dinamico quando l&#39;immagine viene inserita nel contenuto.

## Problemi noti

**Problema**: La `[magento_root]/index.php` Il file è stato rimosso e l&#39;applicazione ora viene eseguita da `/pub` per impostazione predefinita per le configurazioni di Apache. Gli archivi gestiti dalle sottocartelle non funzioneranno come previsto e potrebbero presentare errori 404. **Soluzione alternativa**: Utilizza i collegamenti simbolici per emulare l’installazione in una sottocartella. Nell&#39;esempio seguente viene utilizzato `https://shop01.com/shop/` per illustrare l’utilizzo di un collegamento simbolico per emulare un’installazione in una sottocartella.

1. Creare una sottodirectory per `https://shop01.com/shop/`:

   ```bash
   mkdir magento_root/pub/shop01
   ```

   ```bash
   cd magento_root/pub/shop01
   ```

1. Crea collegamenti simbolici per le directory principali dello store nella directory appena creata:

   ```bash
   ln -s ../media media
   ```

   ```bash
   ln -s ../static static
   ```

   ```bash
   ln -s ../../../pub pub
   ```

   ```bash
   ln -s ../.htaccess .htaccess
   ```

   ```bash
   ln -s ../health_check.php health_check.php
   ```

1. Crea un `index.php` all&#39;interno della nuova directory (`magento_root/pub/shop01/index.php`) e aggiungi il seguente contenuto:

```php
   <?php
   require realpath(__DIR__) . '/../../../app/bootstrap.php';

   switch ($_SERVER['HTTP_HOST']) {
       case 'shop01.com':
       case 'www.shop01.com':
           $params = $_SERVER;
           $params[\Magento\Store\Model\StoreManager::PARAM_RUN_CODE] = 'shop01';
           $params[\Magento\Store\Model\StoreManager::PARAM_RUN_TYPE] = 'website';
           $bootstrap = \Magento\Framework\App\Bootstrap::create(BP, $params);
           $app = $bootstrap->createApplication(\Magento\Framework\App\Http::class);
           $bootstrap->run($app);
           break;

       default:
           $bootstrap = \Magento\Framework\App\Bootstrap::create(BP, $_SERVER);
           /** @var \Magento\Framework\App\Http $app */
           $app = $bootstrap->createApplication(\Magento\Framework\App\Http::class);
           $bootstrap->run($app);
       break;
   }
```

1. Configura il server Apache in modo che punti alla nuova sottodirectory. Le configurazioni di distribuzione possono variare notevolmente. Ecco un esempio di configurazione del server:

```xml
   <VirtualHost *:80>
       DocumentRoot "magento_root/pub/shop1"
       ServerName shop01.com
   </VirtualHost>
```

Lo strumento di migrazione dei dati consente di trasferire i dati dell’archivio Magento 1.x esistenti a Magento 2.x. Questa interfaccia a riga di comando include funzioni di verifica, tracciamento dell’avanzamento, registrazione e test. Per le istruzioni di installazione, consulta [Installare lo strumento di migrazione dei dati](../../../tools/data-migration-tool/install.md). Prendere in considerazione l&#39;esplorazione o il contributo al [Archivio di migrazione dei dati](https://github.com/magento/data-migration-tool).

La [Toolkit di migrazione del codice](https://github.com/magento-commerce/code-migration) aiuta a trasferire estensioni e personalizzazioni dell’archivio Magento 1.x esistenti a Magento 2.x. L&#39;interfaccia della riga di comando include script per la conversione di moduli e layout di Magento 1.x.
