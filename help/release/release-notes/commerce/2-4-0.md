# Note sulla versione di Adobe Commerce 2.4.0

Adobe Commerce 2.4.0 introduce il supporto per PHP 7.4, Elasticsearch 7.6.x e MySQL 8.0. Modifiche sostanziali alla sicurezza includono l&#39;abilitazione dell&#39;autenticazione a due fattori nell&#39;amministratore per impostazione predefinita. I miglioramenti B2B includono il nuovo flusso di lavoro Approvazione ordine .

**Con questa versione, le integrazioni dei metodi di pagamento Authorize.Net, eWay, CyberSource e Worldpay sono state rimosse dal codice core. I commercianti devono migrare alle estensioni ufficiali disponibili nella Commerce Marketplace**.

>[!NOTE]
>
>**Integrazione Braintree pagamento**: Prima della versione 2.4.0, si consigliava agli esercenti di installare e configurare l’estensione ufficiale dell’integrazione dei pagamenti Braintree dalla Commerce Marketplace per sostituire l’integrazione di base. Con questa versione (2.4.0), l’estensione è ora inclusa nella versione. I commercianti devono seguire ulteriori passaggi per garantire che Braintree funzioni correttamente in una distribuzione 2.4.0. Vedi [Braintree](https://docs.magento.com/user-guide/payment/braintree.html) per ulteriori informazioni su come migrare alla versione 2.4.0.

Questa versione include tutti i miglioramenti alla qualità di base inclusi nelle versioni 2.3.5-p1, oltre 100 nuove correzioni al codice di base e 30 miglioramenti alla sicurezza. Include la risoluzione di 226 questioni GitHub da parte dei membri della community. Questi contributi della community spaziano dalla pulizia secondaria del codice di base ai miglioramenti significativi in Inventory management e GraphQL.

>[!NOTE]
>
>Le versioni minori apportano miglioramenti sostanziali al codice. Prima di eseguire l’aggiornamento alla versione 2.4.0, verifica che l’ambiente soddisfi il minimo [requisiti tecnici in materia di stack](../../../installation/system-requirements.md).

## Applica `AC-3022.patch` continuare ad offrire DHL come vettore marittimo

DHL ha introdotto la versione 6.2 dello schema e a settembre 2022 diventerà obsoleta la versione 6.0 dello schema. Adobe Commerce 2.4.4 e versioni precedenti che supportano solo la versione 6.0 dell’integrazione DHL. Si applicano i merchants che distribuiscono queste versioni `AC-3022.patch` al più presto per continuare ad offrire DHL come vettore marittimo. Consulta la sezione [Applicare una patch per continuare ad offrire DHL come vettore di spedizione](https://support.magento.com/hc/en-us/articles/7707818131597-Apply-a-patch-to-continue-offering-DHL-as-shipping-carrier) Articolo della Knowledge Base per informazioni sul download e l&#39;installazione della patch.

## Patch di sicurezza disponibile

Ora i merchants possono installare correzioni di sicurezza che richiedono tempi di esecuzione, senza applicare le centinaia di correzioni funzionali e i miglioramenti forniti da una versione trimestrale completa (ad esempio, 2.3.5-p2). La patch 2.3.5.2 (pacchetto compositore 2.3.5-p2) è una patch di sicurezza che fornisce correzioni per le vulnerabilità che sono state identificate nella nostra precedente versione trimestrale, 2.3.5-p1. Tutte le hotfix applicate alla versione 2.3.5 sono incluse in questa patch di sicurezza. A *hotfix* fornisce una correzione a una versione rilasciata che risolve un problema o un bug specifico.)

Per informazioni generali sulle patch di sicurezza, vedi [Introduzione alla nuova versione della patch di sicurezza](https://community.magento.com/t5/Magento-DevBlog/Introducing-the-New-Security-Patch-Release/ba-p/141287).Per istruzioni su come scaricare e applicare patch di sicurezza (compresa la patch 2.3.5-p2), vedi [Installazione rapida in locale](../../../installation/composer.md). Le patch di sicurezza includono solo correzioni di bug di sicurezza, non i miglioramenti di sicurezza aggiuntivi inclusi nella patch completa.

Anche se il codice per queste funzioni è in bundle con le versioni trimestrali , diversi di questi progetti (ad esempio, B2B, Page Builder, Inventory management e Progressive Web Application (PWA) Studio) vengono rilasciati anche in modo indipendente. Le correzioni di bug per questi progetti sono documentate nelle informazioni sulla versione separate e specifiche per il progetto, disponibili nella documentazione di ciascun progetto.

### Miglioramenti sostanziali della sicurezza

Questa versione include oltre 30 correzioni di sicurezza e miglioramenti alla sicurezza della piattaforma.

#### Oltre 30 miglioramenti a livello di sicurezza per la chiusura di vulnerabilità RCE (Remote Code Execution) e XSS (cross-site scripting)

Finora non si sono verificati attacchi confermati relativi a tali problemi. Tuttavia, alcune vulnerabilità possono essere potenzialmente sfruttate per accedere alle informazioni sui clienti o per prendere il controllo delle sessioni di amministrazione. La maggior parte di questi problemi richiede che un autore dell&#39;attacco ottenga prima l&#39;accesso all&#39;amministratore. Di conseguenza, ti ricordiamo di prendere tutte le misure necessarie per proteggere il tuo amministratore, compresi, tra l’altro, questi sforzi: inserire nell&#39;elenco Consentiti IP, [autenticazione a due fattori](https://devdocs.magento.com/guides/v2.4/security/two-factor-authentication.html), l&#39;utilizzo di una VPN, l&#39;utilizzo di una posizione unica anziché `/admin`e una buona igiene delle password. Vedi [Aggiornamenti di sicurezza disponibili per il Magento](https://helpx.adobe.com/security/products/magento/apsb20-47.html) per una discussione su questi problemi fissi.

#### Ulteriori miglioramenti della sicurezza

* **Implementazione di 2FA per account amministratore, account utente magento.com e accesso Cloud SSH**

   * **Protezione dell’account amministratore**. L’autenticazione a due fattori (2FA) è ora necessaria per l’amministratore. Gli utenti amministratori devono prima configurare i propri 2FA prima di accedere all’amministratore tramite l’interfaccia utente o un’API web. 2FA è abilitato per impostazione predefinita. Si consiglia vivamente di non disabilitare il modulo 2FA.  Questo passaggio aggiuntivo di autenticazione rende più difficile per gli utenti malintenzionati accedere all’amministratore senza autorizzazione. Vedi [Autenticazione a due fattori (2FA)](https://devdocs.magento.com/guides/v2.4/security/two-factor-authentication.html). <!-- MC-22631-->

   * **Protezione dell&#39;account**. L’autenticazione a due fattori (2FA) fornisce un livello di sicurezza aggiuntivo e opzionale per proteggere meglio il tuo account magento.com da utenti non autorizzati che potrebbero voler usare il tuo account in modi che non desideri. Vedi [Protezione dell&#39;account](https://docs.magento.com/user-guide/magento/magento-account-secure.html).

   * **Protezione dell’accesso all’SSH di Cloud**. L’Adobe sull’infrastruttura cloud fornisce l’imposizione dell’autenticazione a più fattori (MFA) per gestire i requisiti di autenticazione per l’accesso SSH agli ambienti Cloud. L’autenticazione a più fattori per 2FA non è abilitata per impostazione predefinita su un progetto.  Adobe consiglia vivamente di abilitare questa funzione. Contatta il supporto per ricevere assistenza. Vedi [Abilitare l’autenticazione a più fattori per l’accesso SSH](https://devdocs.magento.com/cloud/project/project-enable-mfa-enforcement.html).

* **La modalità rigorosa del filtro modelli è ora attivata per impostazione predefinita**. I componenti (comprese le pagine e i blocchi CMS) che utilizzano il filtro modelli in modalità legacy possono essere vulnerabili all’esecuzione di codice remoto (RCE, Remote Code). L’abilitazione della modalità rigorosa per impostazione predefinita garantisce che gli attacchi RCE non possano essere attivati deliberatamente. <!-- MC-22982-->

* **Il rendering dei dati per i provider di dati dell’interfaccia utente è ora disattivato per impostazione predefinita**. Questo elimina l’opportunità per gli utenti dannosi di eseguire JavaScript arbitrari. <!-- MC-17356-->

* **Nuovo `\Magento\Framework\Escaper` Classe**. Questa classe è fornita per `.phtml` modelli e classi PHP responsabili della generazione di HTML. Questa classe contiene metodi di pulizia di HTML relativi a più contesti. La `$escaper` la variabile locale è disponibile all&#39;interno di `.phtml` modelli e devono essere utilizzati al posto della pagina obsoleta `$block->escape{method}`. Utilizzo `$escaper` piuttosto che `$block` come uso `$block->escape{method}` è stato dichiarato obsoleto.

* **Supporto per il file security.txt**. Questo file è un file standard di settore sul server che aiuta i ricercatori della sicurezza a segnalare potenziali problemi di sicurezza agli amministratori del sito.

* **Miglioramenti al supporto dei criteri sulla sicurezza dei contenuti (CSP)**. `SecureHtmlRenderer` è stato aggiunto al framework ed è disponibile in `.phtml` modelli per inserire nella whitelist in linea `style` e `script` tag. Gli script e gli stili in linea non sono consentiti con la configurazione CSP predefinita, che può essere ignorata dai file XML. <!-- MC-34433-->

### Aggiornamenti della piattaforma

I seguenti aggiornamenti della piattaforma consentono di migliorare la sicurezza e le prestazioni del sito web. Le versioni supportate di PHP e PHPUnit, Elasticsearch, MySQL e altre dipendenze sono elencate in [requisiti dello stack tecnologico](../../../installation/system-requirements.md).

* **Supporto PHP 7.4 introdotto e PHP 7.1 e 7.2 obsoleto**. 2.4.0 introduce il supporto per PHP 7.4.

* **Supporto per PHPUnit 9.x e rimozione di PHPUnit 6.5**. PHP 7.4 richiede l&#39;uso dell&#39;ultimo framework di test PHPUnit, che è PHPUnit 9.x. I fornitori di estensioni Commerce Marketplace devono confermare che tutte le nuove versioni di estensione sono compatibili con PHP 7.4 e che tutti i test di unità e integrazione sono stati configurati per essere eseguiti con PHPUnit 9.

* **Supporto per Elasticsearch 7.6.x**. Elasticsearch 7.6.x è ora il motore di ricerca del catalogo predefinito per Adobe Commerce e Adobe Commerce. Non è possibile installare o aggiornare alla versione 2.4.0 senza installare anche Elasticsearch 7.6.x. Il codice Elasticsearch versione 2.x è stato rimosso. Le versioni 5.x e 6.x di Elasticsearch sono state dichiarate obsolete e non sono più supportate. Vedi [Prerequisiti per i motori di ricerca](../../../installation/prerequisites/search-engine/overview.md).

* **Supporto di MySQL 8.0**. La versione 2.4.x supporta MySQL 8.x. (2.4.0 è stato testato con MySQL 8.0.20.) I commercianti sono invitati a migrare le implementazioni in MySQL 8.x per sfruttare le prestazioni, la sicurezza e l&#39;affidabilità migliorate. Anche se MySQL 5.7 è ancora supportato per Magento 2.4.x, MySQL 5.6 non è più supportato. Non è possibile ospitare 2.4.x con un database MySQL 5.6. Vedi [MySQL](../../../installation/prerequisites/database/mysql.md).

* **Supporto per MariaDB 10.4**. Il supporto per MySQL 8.0 offre agli esercenti l&#39;opportunità di distribuire MariaDB 10.4 con Adobe Commerce. Anche se i commercianti possono ancora utilizzare MariaDB 10.2 con 2.4.0, si consiglia di eseguire l&#39;aggiornamento a MariaDB 10.4 per migliorare le prestazioni e l&#39;affidabilità. MariaDB 10.0 e 10.1 non sono più supportati (a causa della rimozione del supporto per MySQL 5.6 in questa versione).
* **Rimozione del motore di ricerca del catalogo MySQL**. Il motore di ricerca MySQL è stato rimosso da 2.4.0 e sostituito come motore di ricerca predefinito con Elasticsearch. Elasticsearch offre funzionalità di ricerca superiori e ottimizzazioni delle prestazioni del catalogo.  Per installare e distribuire la versione 2.4.0 di tutti i merchant è necessario disporre di Elasticsearch. Vedi [Verifica che l&#39;Elasticsearch sia installato](https://experienceleague.adobe.com/docs/commerce-operations/upgrade-guide/prepare/prerequisites.html).

* **Migrazione delle dipendenze da Zend Framework a [Progetto Laminas](https://getlaminas.org/about/foundation)** per riflettere la transizione di Zend Framework al progetto Laminas della Linux Foundation. Zend Framework è stato dichiarato obsoleto. Consulta la sezione [Migrazione del quadro di Zend al progetto Laminas](https://community.magento.com/t5/Magento-DevBlog/Migration-of-Zend-Framework-to-the-Laminas-Project/ba-p/443251) Post DevBlog.

* **Decomposizione dei titolari del trattamento** consente agli sviluppatori di estensioni di implementare ActionInterface direttamente senza le classi &quot;layer supertype&quot;. Consulta la sezione [Decomposizione dei titolari del trattamento](https://community.magento.com/t5/Magento-DevBlog/Decomposition-of-Magento-Controllers/ba-p/430883) Post DevBlog. _Miglioramento avviato da Vinai Kopp nella richiesta di pull [16268](https://github.com/magento/magento2/pull/16268) e messo a punto da Lukasz Bajsarowicz su richiesta pull [26778](https://github.com/magento/magento2/pull/26778)_. [GitHub-9582](https://github.com/magento/magento2/issues/9582)

* **Rimozione dell&#39;integrazione di base del codice di protezione contro le frodi Signifyd**. Questa funzionalità di base non è più supportata. I commercianti dovrebbero migrare [Estensione Signifyd Fraud &amp; Chargeback Protection](https://marketplace.magento.com/signifyd-module-connect.html) disponibile in Commerce Marketplace.

* Il browser Internet Explorer 11.x non è più supportato.

### Miglioramenti delle prestazioni

* **Miglioramenti alla logica di invalidazione della sezione dati cliente**. Questa versione introduce un nuovo modo di invalidare tutti i dati delle sezioni cliente che evita un problema noto con l’archiviazione locale quando questa è personalizzata `sections.xml` gli invalidamenti sono attivi.  (In precedenza, il contenuto privato (archiviazione locale) non veniva popolato correttamente quando si utilizzava un *etc/frontend/sections.xml* con invalidazioni azioni azioni). Vedi [Contenuto privato](https://developer.adobe.com/commerce/php/development/cache/page/private-content/).

* **Ottimizzazioni multiple per le prestazioni di Redis**. I miglioramenti riducono al minimo il numero di query a Redis eseguite su ogni richiesta. Queste ottimizzazioni includono:

   * Diminuzione delle dimensioni dei trasferimenti di dati di rete tra Redis e Magento

   * Riduzione del consumo di CPU da parte di Redis migliorando la capacità dell&#39;adattatore di determinare automaticamente ciò che deve essere caricato

   * Riduzione delle condizioni di gara sulle operazioni di scrittura di Redis
   Vedi [Usa Redis per la pagina e la cache predefinita](../../../configuration/cache/redis-pg-cache.md) e [Configurare la memorizzazione in cache](../../../configuration/cache/caching-overview.md).

* **È stata migliorata la memorizzazione in cache dei risultati delle query SQL nelle tabelle di inventario**. Questi miglioramenti includono:

   * Memorizzazione in cache delle query SQL nel `inventory_stock_sales_channel` tabella (1 query invece di 16)

   * Memorizzazione in cache del risultato delle query nel `inventory_stock` tabella (1 query invece di 16)

* **Miglioramento fino al 25-30% delle prestazioni del componente aggiuntivo per carrello dell&#39;ordine rapido**.

* Gli esercenti ora possono utilizzare [carico lento](https://en.wikipedia.org/wiki/Lazy_loading) per caricare le immagini.

## Kit di strumenti di migrazione

Lo strumento di migrazione dei dati consente di trasferire i dati dell’archivio Magento 1.x esistenti a Magento 2.x. Questa interfaccia a riga di comando include funzioni di verifica, tracciamento dell’avanzamento, registrazione e test. Per le istruzioni di installazione, consulta [Installare lo strumento di migrazione dei dati](https://experienceleague.adobe.com/docs/commerce-operations/tools/data-migration/basics/install.html). Prendere in considerazione l&#39;esplorazione o il contributo al [Archivio di migrazione dei dati](https://github.com/magento/data-migration-tool).

La [Toolkit di migrazione del codice](https://github.com/magento/code-migration) aiuta a trasferire estensioni e personalizzazioni dell’archivio Magento 1.x esistenti a Magento 2.x. L&#39;interfaccia della riga di comando include script per la conversione di moduli e layout di Magento 1.x.
